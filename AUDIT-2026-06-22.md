# Naina's Baby Registry — Fresh Deep-Dive Audit

_Audit date: 22 Jun 2026 · against app `v30` · scope: code & architecture, UX & interactions, visual design, data pipeline, accessibility · method: full source read + live diagnosis on `https://tracker-aj.vercel.app` via a real browser (read-only — no shared data changed)._

This audit was run **independently** (without leaning on the previous `AUDIT.md`), then reconciled against it at the end (§9). Short version: the previous audit was largely **actioned** — most of its security and polish findings are fixed in v30. What's left is a smaller set of **data-integrity bugs, one dormant core feature, and a content-quality gap** that quietly undercuts the registry's main job.

Each finding carries a **confidence** (0–1) and is tagged **✅ verified live** where I confirmed it in the running app.

---

## 0. Verdict first

**The app does most of what it sets out to do, and the craft is high.** The friend/admin split works, the visual system is genuinely tasteful, dark mode and the modal system are polished, and there were **zero console errors** across every view I exercised. The security model has been properly rebuilt since the last audit (Edge Function + RLS + SRI + CSP).

**But two of its three headline promises are only partly delivered:**

1. The **"deal tracker"** currently surfaces **0 deals** (verified live) — the feature is effectively dormant.
2. The **registry links** mostly point at **search-result pages, not exact products** (407 of 412 links), which contradicts the project's own design principle and weakens both the family experience and the price pipeline.

Plus one **silent data-loss bug** in Export/Import that I'd fix today (I have — see §8).

**Overall: 7.5/10.** Strong bones; the gaps are about *value realization and data integrity*, not craft. _Confidence: 0.85._

## 1. Top 5 to fix first

1. **Export/Import silently drops manual prices, deal targets, and bought-counts** — a data-loss bug in the one feature you'd use to recover data. _(§2.1, P0 — **fixed in this pass**)_
2. **The deal engine produces nothing** (`0 Deals` live). Without price history or targets, `current == average`, so nothing ever flags. _(§4.1, P1)_
3. **407/412 product links are searches, not exact products** — family clicks "View" and lands on a search page; the price job can't read them either. _(§4.2 / §5.1, P1)_
4. **Concurrent edits can be silently lost** — `pushState` overwrites the shared row wholesale, and a `409` (stale) response makes the client **discard its own pending change**. The merge code to prevent this exists but is wired nowhere. _(§2.2, P1 — **partial fix in this pass**)_
5. **Registry shows confusing near-duplicate cards** — one item with 4 finalised options renders as 4 cards, each titled with a cryptic internal nickname. _(§3.1, P2)_

## 2. Code & architecture

The structure is clean: `index.html` (shell + design system), `app.js` (all logic), `products.json` (catalogue), with tidy event-delegation. The security rebuild is real and correct in shape (server-side password check in the Edge Function; anon key only reads; SRI on both CDN scripts; a sensible CSP in `vercel.json`). Remaining issues:

### 2.1 — P0 · Export/Import is an incomplete backup → data loss ✅ verified in code
`exportEdits()` serializes only `{track, hidden, useritems, pins, statusovr, useropts, userqty}`. It **omits** `userprices`, `usertarget`, and `userbought` — even though `localState()` (the thing actually synced) includes them. So:
- "Export edits" produces an incomplete backup.
- "Import" calls `applyShared()`, which sets the missing keys to `{}` — **wiping** any manual prices, deal targets, and bought-counts you had, then syncing that wipe to every device.

This is the highest-severity *correctness* bug in the app. **Fixed in this pass** by exporting the full `localState()` (§8). _Confidence: 0.97._

### 2.2 — P1 · Push clobbers instead of merges; `409` discards your edit ✅ verified in code
`pushState()` POSTs `localState()` wholesale. The Edge Function adds optimistic concurrency (rejects writes older than stored, `409`), which is good — but the client's `409` handler calls `pullAndRender()`, which **adopts remote and drops the local change that was mid-flight.** Net effect for two admins editing close together: one edit silently vanishes.

You already have `mergeState()` / `mItems` / `mOpts` / `mPins` / `mO` — but they are **dead code** (defined, never called), and `mergeState` itself omits `userprices`/`usertarget`/`userbought`. **Partial fix in this pass:** I extended the merge helpers to cover every key and wired a **merge-then-retry on `409`** so a stale write reconciles instead of discarding. Please test with two devices before relying on it (§8 caveat). _Confidence: 0.9._

### 2.3 — P2 · Dead code & maintainability
- The five merge helpers were unused (now used per §2.2).
- `app.js` is 485 lines of dense, single-expression one-liners with no `package.json`, lint, or tests. At this size it's manageable, but a smoke test over `priceInfo()`, `mergeState()`, and the deal logic would catch regressions cheaply.
- Cache-busting is inconsistent and manual: `app.js?v=30`, `products.json?v=5`, `proscons.json` un-versioned — while `vercel.json` already sends `no-cache`. The `?v=` params are redundant and easy to forget to bump. Pick one mechanism. _Confidence: 0.8._

### 2.4 — P2 · Realtime + 5s poll both run
Boot subscribes to Supabase Realtime **and** sets a 5s `setInterval` poll. The poll is a fine fallback, but running both means a steady read every 5s even when Realtime is healthy. Consider widening the poll (e.g. 20–30s) when the Realtime channel is subscribed. _Confidence: 0.75._

### 2.5 — Note · Friend-side boot has no error surface
If `products.json` fails to load, boot sets a message into `#list` and returns early — but `#list` lives in the admin Dashboard, which friends never see. A friend would get the hero + an empty registry with no explanation. Low probability, easy to add a friend-visible fallback. _Confidence: 0.8._

## 3. UX & interactions

The interaction design is good: keyboard-operable cards, a real focus trap, Escape-to-close, debounced search, a sync toast, and an admin "To Buy" cockpit that's genuinely usable. Gaps:

### 3.1 — P2 · Registry duplicate cards are confusing ✅ verified live
When an item has several finalised options (item 1 has 4 pins), the public registry renders **4 separate cards**, each titled with the option's internal nickname — "Next", "Brown Pudding First Size", "Brown Reindeer 3months", "Next" — and subtitled with the same parent item name ("Thermal/zip footed sleepsuits"). To a family member this looks like four unrelated products. Two options:
- **Group** multiple finalised options under one item card ("Sleepsuits — 4 picks"), or
- **Constrain** the registry to one finalised pick per item (multi-finalise is an admin research tool, not a family-facing concept).

Either is better than today's repetition. _Confidence: 0.85._

### 3.2 — P2 · Mixed message: "not a wish-list" vs buy-links + prices ✅ verified live
The hero says _"think of this as a cosy little peek… not a wish-list to shop from"_, yet every registry card shows country flag **buy links** and a **₹ price**. Decide which story you're telling: a cosy peek (drop the links/prices on the friend view) or a real registry (lean in, add a soft "let us know you're gifting this"). Right now it's halfway between, which reads slightly contradictory. _Confidence: 0.75._

### 3.3 — P2 · Card is `role="button"` but contains buttons/links
Cards use `role="button" tabindex="0"` while nesting interactive controls (delete, "Buy best", "Details"). Nesting interactive elements inside a button role is invalid and can confuse screen readers (and the whole card announces as one big button). Consider making the image/title a single focusable target and leaving the inner controls as siblings, rather than wrapping everything in a button role. _Confidence: 0.8._

### 3.4 — Note · Mobile breakpoints not fully verifiable live
The browser tool kept a ~1480px content area even after resizing, so I couldn't exercise the phone breakpoints in the running app. The CSS itself has the right media queries (hero at 620px, compare grid at 560px, button-row wrap at 600px, modal header stacking at 480px), so this is "unverified," not "broken." Worth a real-device pass since a registry is a phone-first artifact. _Confidence: 0.6 (unverified)._

## 4. Data & price pipeline (the "deal tracker" half)

### 4.1 — P1 · The deal engine surfaces nothing ✅ verified live
The live Dashboard shows **`0 Deals`**. Three compounding causes:
- Only **11 of 57** items have a `producturl`, so only ~19% can ever be auto-priced.
- Priced items typically have **a single price point**, so `current == average` → the `cur < avg*(1-0.05)` test can never fire.
- Few items have a **target** set, which is the only path that flags a deal without history.

So the headline "deal tracker" is dormant for the people (well, person) it serves. The FX-at-render fix from the last audit is correctly in place, so *when* history accrues this will work — but today there's nothing to show. **Recommendation:** lean into **targets** as the primary deal mechanism (set a target ₹ per item → flag when the cheapest market drops to/below it). It's the only signal that works without long history, and it matches how you actually shop. _Confidence: 0.9._

### 4.2 — P1 · Links are searches, not products ✅ verified live + in data
**407 of 412** option links are search-query URLs (`/s?k=…`, `/search?q=…`), not exact product pages. Consequences:
- Family clicking "View product" lands on a **search results page**, not the item — poor for a registry.
- The price job (which needs a real product page to read JSON-LD/Shopify/meta) can only target the 11 `producturl` items, which is *why* §4.1 is empty.

The DESIGN.md principle was softened to "prefer exact, fall back to scoped search," but in practice it's ~99% fallback. This is the single biggest lever on the registry's real usefulness. **Recommendation:** progressively replace the highest-priority items' search links with exact `producturl`s (the live registry's 8 finalised picks already use exact Next URLs — good — so the pattern exists; extend it). _Confidence: 0.92._

### 4.3 — P2 · Price Log is single-market per item ✅ verified live
The Price Log's premise is cross-market India/UK/Canada comparison, but live, almost every row has a price in **one** market and "—" in the others — so "Best buy" is trivially "the only market with data." The multi-market value only appears if items carry `producturls` for more than one region. Either populate multi-region URLs for a few flagship items, or present the log as "latest known price" rather than a comparison. _Confidence: 0.85._

### 4.4 — P2 · README catalogue count drift
README says **54 base items**; `products.json` has **57**; the live Dashboard shows **58** (incl. a synced user item). Pick one authoritative number. **Updated README to 57 in this pass** (§8). _Confidence: 0.95._

## 5. Visual design

The aesthetic hits the DESIGN.md brief: system font stack, tuned weights, pill radii, layered soft shadows, a restrained blue accent with semantic status colors, and tasteful motion. Dark mode is now correct (the tile-bleed from the last audit is fixed). Minor remaining polish:

### 5.1 — P2 · `object-fit: cover` crops packaging
Product images use `cover`, which can crop labels/products (visible on the boxed bottle/diaper shots). For a shopping context, `contain` on the tile background preserves the whole product. Tradeoff — `cover` looks tidier, `contain` is more honest. Consider `contain` where product identity matters (modal `.optimg`, compare thumbnails). _Confidence: 0.65._

### 5.2 — P2 · Redundant "finalised" signalling
In the item modal, a finalised option shows both a "★ FINALISED" rank chip **and** a "★ Finalised" button — two near-identical signals in one row. Collapse to one. _Confidence: 0.7._

### 5.3 — Note · Type scale is bespoke but consistent
A spread of hand-tuned weights (740/680/620/590/560) and sizes (16.5 vs 15px titles) reads intentional and looks fine; a documented token scale would lock it down, but this is taste-level, not a defect. _Confidence: 0.7._

## 6. Security & data integrity (re-checked, not the focus)

The model is now sound **in shape**: anon key reads only, writes go through the `save-state` Edge Function that checks a server-side `ADMIN_PASSWORD`, payload-size capped, constant-time-ish compare, optimistic concurrency. SRI + CSP are in place. Two caveats I **could not** verify from outside (and didn't attempt writes against your live DB):

- **RLS must actually be enabled.** The whole model depends on the SQL in `SECURITY-SETUP.md` §1 having been run. If it wasn't, the row is world-writable again. Verify: logged-out, a direct `POST` to `/rest/v1/tracker_state` should return 401/403. _Confidence: 0.6 (unverified)._
- **`state-backup.json` is committed to a public repo.** It's your family's registry state (pins, items, notes, prices) — not credentials, but it is personal data sitting in public. Consider a private gist or a repo-secret-gated artifact instead. _Confidence: 0.9._

(These were not in your top-3 priorities, so I'm flagging, not fixing.)
